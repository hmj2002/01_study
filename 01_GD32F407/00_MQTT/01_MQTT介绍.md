## 1.2 开发分工说明

围绕物联网点灯流程，核心分工可概括为：

### 📱 前端（App / Web）—— 管"面子"（人机交互）

| 项目         | 说明                                                         |
| ------------ | ------------------------------------------------------------ |
| **做什么**   | 画出控制界面（如开关按钮）                                   |
| **怎么交互** | 点击时发送 HTTP 请求给后端；通过 WebSocket 接收新状态并刷新界面 |

### 🏢 后端（服务器）—— 管"脑子"（业务与中转）

| 项目         | 说明                                                         |
| ------------ | ------------------------------------------------------------ |
| **做什么**   | 处理用户权限、逻辑判断与数据流转                             |
| **怎么交互** | 收到前端请求后，调用华为云 SDK 下发指令；收到云端设备状态反馈后，转发给前端 |

### 📟 嵌入式（ESP32 / 硬件端）—— 管"身子"（硬件执行）

| 项目         | 说明                                                         |
| ------------ | ------------------------------------------------------------ |
| **做什么**   | 写底层固件，让芯片联网并控制硬件                             |
| **怎么交互** | 通过 MQTT 维持与云端长连接；解析"开灯"指令驱动 GPIO 引脚让灯亮起，并上报真实硬件状态 |

---

### 一句话总结

&gt; 前端发命令看状态，后端居中调度做中转，嵌入式底层连网控硬件。

---

### 用"手机点外卖"来理解

| 角色             | 对应                 | 核心任务       | 大白话                                                       |
| ---------------- | -------------------- | -------------- | ------------------------------------------------------------ |
| 📱 前端（App）    | 外卖 App 界面        | 负责看和点     | 画出"下单"按钮。你点一下，它把命令发走；等外卖好了，界面自动刷新，告诉你"已送达" |
| 🏢 后端（服务器） | 平台总部系统         | 负责算和传     | 收到你的下单请求，先查你有没有扣款成功（对权限），然后把任务精准派发给具体的商家（中转指令） |
| 📟 嵌入式（硬件） | 实体店里的自动制茶机 | 负责联网和通电 | 通电联网的真实机器。接到总部派单，内部电机和阀门立刻通电开动（驱动 GPIO）把奶茶做出来（硬件灯亮），并反馈"已做好" |

---

### 🔄 三步闭环流程

1. **点单**：前端（App）按按钮 ➡️ 后端（总部）收到请求
2. **做单**：后端（总部）下发任务 ➡️ 嵌入式（制茶机）通电死磕硬件，灯亮起
3. **反馈**：嵌入式（制茶机）上报"做好了" ➡️ 后端转达 ➡️ 前端（App）界面刷新

## 1.3 控制与反馈的核心闭环

![img](https://cdn.nlark.com/yuque/__mermaid_v3/93155600d01168bbdcbcb66e9592ab63.svg)

1. **指令下发（Control Link）**：手机 App 发出控制请求，由后端转发或直接通过云平台，利用 MQTT 的长连接特性将指令精准推送到指定的终端设备，触发硬件引脚电平变化。

2. **状态反馈（Feedback Link）**：设备执行动作后，必须将真实的硬件状态异步上报至云端，再由云端同步回 App 界面。只有看到状态回显，才算完成一次完整的物联网控制交互。

# 2. MQTT 协议

## 2.1 简介

MQTT（Message Queuing Telemetry Transport）是一种轻量级的、基于**发布/订阅（Publish/Subscribe）**模式的通信协议。它构建于 TCP/IP 协议之上，专为低带宽、高延迟、网络不稳定的物联网环境设计。

### 核心技术优势

| 优势         | 说明                                                         |
| ------------ | ------------------------------------------------------------ |
| **极致轻量** | 固定头部最小仅占 2 字节，最大程度降低了报头开销，非常省流量和电量 |
| **低耦合度** | 发布者与订阅者在时间、空间和业务逻辑上完全解耦，均不需要知道对方的存在，全靠消息代理（Broker）中转 |
| **双向通信** | 设备既可以作为发布者上报数据，也可以作为订阅者接收控制指令   |

---

## 2.2 服务质量等级 (QoS)

MQTT 通过服务质量（QoS）来平衡传输可靠性与网络开销，共有三个等级：

| QoS 等级  | 机制名称 | 行为特点                                                     | 典型适用场景                                                 |
| --------- | -------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **QoS 0** | 最多一次 | 消息"尽力而为"地发送，不进行确认，可能会丢失。发送速度最快，开销最小 | 周期性且允许丢失的数据（如每秒更新一次的环境温度、车载时速） |
| **QoS 1** | 至少一次 | 保证消息必须送达，但由于重发机制，接收方可能会收到重复消息   | 核心控制指令（如本节的"开灯"命令，必须确保设备收到）         |
| **QoS 2** | 仅有一次 | 通过四阶段握手，确保消息不仅送达，而且绝不重复。开销和延迟最大 | 对准确性要求极高的场景（如计费系统、支付确认、设备解绑）     |

### MQTT 的三位核心角色

| 角色                        | 说明                                                         | 示例           |
| --------------------------- | ------------------------------------------------------------ | -------------- |
| **消息发布者 (Publisher)**  | 数据的生产者，只负责把数据发送到特定主题上，不关心谁会接收   | 温度传感器     |
| **消息代理 (Broker)**       | 整个系统的通信中枢，接收所有发布者的消息，根据主题转发给订阅者 | 华为云 IoT     |
| **消息订阅者 (Subscriber)** | 数据的消费者，提前订阅感兴趣的主题，有新消息时立刻收到推送   | 手机 App、电脑 |

### 通俗比喻：微信公众号机制

![img](https://cdn.nlark.com/yuque/__mermaid_v3/879c94aad50831ff4dcd1856beb97c94.svg)

| MQTT 操作            | 对应公众号动作 | 说明                                                         |
| -------------------- | -------------- | ------------------------------------------------------------ |
| **订阅 (Subscribe)** | ➡️ 关注公众号   | 你和朋友提前关注了"天气预报"这个公众号，这个动作被微信服务器记录 |
| **发布 (Publish)**   | ➡️ 发布文章     | 公众号运营者（传感器）写完"今日气温：28°C"的文章，点击发布，文章被发给微信服务器（Broker） |
| **分发**             | ➡️ 推送文章     | 微信服务器收到文章后，查找关注者列表，发现你和朋友都在其中，于是立刻把文章推送到你们的手机上 |
